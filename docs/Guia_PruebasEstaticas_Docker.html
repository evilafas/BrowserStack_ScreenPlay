Introducción y Objetivos

🔍 1.1 ¿Qué son las Pruebas Estáticas?

Las pruebas estáticas (o análisis estático de código) son técnicas de verificación que inspeccionan el código fuente sin ejecutarlo. A diferencia de las pruebas dinámicas (que ejecutan el sistema), el análisis estático detecta defectos en la fase de desarrollo, reduciendo el coste de corrección hasta 10 veces respecto a los encontrados en producción.

En el contexto del Gobierno de Automatización Telefónica, el análisis estático es una capa obligatoria de calidad que complementa las pruebas funcionales BDD de Serenity.

💡
¿Por qué análisis estático en proyectos de automatización?

El código de automatización es código de producción en cuanto a mantenibilidad. Los mismos defectos de diseño (código duplicado, complejidad innecesaria, vulnerabilidades) que afectan al software de negocio también afectan a los frameworks de pruebas.

🛠️ 1.2 Herramientas del Stack

📐 Checkstyle
v10.17.0

Verifica que el código sigue las convenciones de estilo definidas por el gobierno Telefónica: nomenclatura PascalCase/camelCase/UPPER_SNAKE_CASE, longitud de líneas, espaciado, imports.

🔬 PMD
v7.5.0

Analiza el AST (árbol sintáctico) del código fuente para detectar: código muerto, duplicaciones, complejidad ciclomática elevada, malas prácticas y vulnerabilidades de seguridad.

🐛 SpotBugs
v4.8.6

Analiza el bytecode compilado (.class) para detectar patrones de bugs conocidos: null pointer exceptions, recursos no cerrados, condiciones de carrera, inyección SQL.

📊 SonarQube
v10.4 Community

Plataforma centralizada de calidad. Agrega resultados de todas las herramientas, calcula deuda técnica, gestiona Quality Gates y muestra tendencias históricas. Corre en Docker.

Configuración de Herramientas de Análisis

📁
Estructura de configuración creada:

config/checkstyle/checkstyle.xml · config/checkstyle/suppressions.xml · config/pmd/ruleset.xml · config/spotbugs/exclude.xml

📐 3.1 Checkstyle — Estilo de Código

Checkstyle verifica que el código fuente cumple con las convenciones del Gobierno de Automatización Telefónica. La configuración principal se encuentra en config/checkstyle/checkstyle.xml.

Reglas principales configuradas:

Categoría Regla Descripción
Nomenclatura TypeName Clases en PascalCase
Nomenclatura MethodName Métodos en camelCase
Nomenclatura ConstantName Constantes en UPPER_SNAKE_CASE (TXT_, BTN_, INPUT_...)
Imports AvoidStarImport Prohibidos imports con .*
Formato LineLength Máximo 150 caracteres por línea
Buenas prácticas Regexp Prohibido System.out/err.print (usar SLF4J)
Llaves NeedBraces Siempre usar llaves en bloques if/for/while

Supresiones (suppressions.xml):

Las siguientes áreas tienen reglas relajadas intencionalmente:

  • src/test/ — Sin requisito de Javadoc
  • *Runner.java — Líneas largas permitidas (@CucumberOptions)
  • *UI.java — Líneas largas permitidas (Targets de Screenplay)
  • UserData.java — Clase Lombok, reglas generadas

🔬 3.2 PMD — Análisis de Código Fuente

PMD analiza el árbol de sintaxis abstracta (AST) del código para detectar patrones problemáticos. El conjunto de reglas en config/pmd/ruleset.xml está adaptado al patrón Screenplay.

Conjuntos de reglas activos:

  • activo bestpractices — Mejores prácticas de Java
  • activo codestyle — Estilo y convenciones
  • activo design — Principios de diseño (SOLID)
  • activo errorprone — Patrones propensos a errores
  • activo security — Vulnerabilidades OWASP
  • activo performance — Problemas de rendimiento
  • activo multithreading — Concurrencia segura
💜
Adaptaciones para Screenplay Pattern

Se excluyen reglas que generarían falsos positivos en el patrón Actor-Task-Interaction: AbstractClassWithoutAbstractMethod, LawOfDemeter (la cadena fluida de Screenplay viola formalmente esta regla pero es el diseño intencionado).

🐛 3.3 SpotBugs — Detección de Bugs en Bytecode

SpotBugs analiza el bytecode compilado (.class) usando más de 400 patrones de bugs conocidos. El filtro de exclusión en config/spotbugs/exclude.xml elimina falsos positivos del framework.

Configuración principal:

Parámetro Valor Descripción
effort max Máximo esfuerzo de análisis (más preciso, más lento)
reportLevel medium Reportar bugs de prioridad media y alta
excludeFilter config/spotbugs/exclude.xml Filtro de falsos positivos

Exclusiones principales configuradas:

  • Clases generadas por Lombok (SE_BAD_FIELD)
  • Tasks de Screenplay — Tasks.instrumented() usa reflexión
  • Excepciones personalizadas — sin serialVersionUID (no necesario en tests)
  • SelfHealingDriverSource — interoperabilidad con Healenium
  • ExcelReader — recursos Apache POI gestionados con try-with-resources

Integración en Gradle

⚙️ 4.1 Actualización de gradle.properties

Se añaden las versiones de los nuevos plugins al archivo gradle.properties:

# ── Análisis estático ──────────────────────────────────────
spotbugsPlugin=6.0.9
sonarPlugin=4.4.1.3373
checkstyleVersion=10.17.0
pmdVersion=7.5.0
spotbugsVersion=4.8.6

🔧 4.2 Configuración de build.gradle

El build.gradle se actualiza con tres bloques principales:

1. Declaración de plugins en buildscript:

buildscript {
    dependencies {
        classpath "net.serenity-bdd:serenity-gradle-plugin:${gradlePlugin}"
        classpath "com.github.spotbugs.snom:spotbugs-gradle-plugin:${spotbugsPlugin}"
        classpath "org.sonarsource.scanner.gradle:sonarqube-gradle-plugin:${sonarPlugin}"
    }
}

apply plugin: 'checkstyle'        // plugin built-in de Gradle
apply plugin: 'pmd'               // plugin built-in de Gradle
apply plugin: 'com.github.spotbugs'  // plugin externo via buildscript
apply plugin: 'org.sonarqube'      // plugin externo via buildscript

2. Configuración de cada herramienta:

// Checkstyle
checkstyle {
    toolVersion = checkstyleVersion
    configFile = file("${project.rootDir}/config/checkstyle/checkstyle.xml")
    configProperties = ['config_loc': "${project.rootDir}/config/checkstyle"]
    maxWarnings = 0
}

// PMD
pmd {
    toolVersion = pmdVersion
    ruleSetFiles = files("${project.rootDir}/config/pmd/ruleset.xml")
    ruleSets = []   // Desactivar ruleset por defecto
}

// SpotBugs
spotbugs {
    toolVersion = spotbugsVersion
    excludeFilter = file("${project.rootDir}/config/spotbugs/exclude.xml")
    effort = 'max'
    reportLevel = 'medium'
}

// SonarQube
sonarqube {
    properties {
        property "sonar.projectKey",   "browserstack-screenplay"
        property "sonar.host.url",     System.getProperty("sonar.host.url", "http://localhost:9000")
        property "sonar.token",        System.getProperty("sonar.token", "")
    }
}

3. Tarea aggregadora:

task staticAnalysis {
    group = 'verification'
    description = 'Ejecuta Checkstyle + PMD + SpotBugs en el código fuente'
    dependsOn checkstyleMain, checkstyleTest, pmdMain, pmdTest, spotbugsMain, spotbugsTest
}

📋 4.3 Tareas Gradle Disponibles

Tarea Descripción Comando
checkstyleMain Checkstyle sobre src/main/java ./gradlew checkstyleMain
checkstyleTest Checkstyle sobre src/test/java ./gradlew checkstyleTest
pmdMain PMD sobre src/main/java ./gradlew pmdMain
pmdTest PMD sobre src/test/java ./gradlew pmdTest
spotbugsMain SpotBugs sobre clases compiladas main ./gradlew spotbugsMain
spotbugsTest SpotBugs sobre clases compiladas test ./gradlew spotbugsTest
staticAnalysis Todas las herramientas a la vez ./gradlew staticAnalysis
sonarqube Enviar análisis a SonarQube ./gradlew sonarqube -Dsonar.token=TU_TOKEN
sonarAnalysis Compilar + analizar + enviar a Sonar ./gradlew sonarAnalysis -Dsonar.token=TU_TOKEN

SonarQube con Docker

🐋 5.1 Arquitectura Docker

Se añaden dos servicios nuevos al docker-compose.yml existente (que ya tenía los servicios de Healenium). Los servicios SonarQube usan su propia red interna y volúmenes nombrados para persistir datos entre reinicios.

Servicio Imagen Puerto Host Descripción
sonarqube-db postgres:14-alpine 5433 Base de datos de SonarQube (separada de Healenium)
sonarqube sonarqube:10.4-community 9000 Servidor SonarQube. UI en http://localhost:9000
⚠️
Importante: vm.max_map_count en Windows/WSL2

SonarQube usa Elasticsearch que requiere vm.max_map_count ≥ 262144. En Windows con Docker Desktop (WSL2), ejecutar en PowerShell como Admin:

wsl -d docker-desktop
sysctl -w vm.max_map_count=262144

Para hacerlo permanente, crear el archivo %USERPROFILE%\.wslconfig:

[wsl2]
kernelCommandLine = sysctl.vm.max_map_count=262144

📄 5.2 Configuración docker-compose.yml (servicios nuevos)

# ═══════════════════════════════════════════════════════════
# SONARQUBE — Plataforma de Calidad de Código
# ═══════════════════════════════════════════════════════════

sonarqube-db:
  image: postgres:14-alpine
  container_name: sonarqube-db
  environment:
    POSTGRES_DB: sonarqube
    POSTGRES_USER: sonar
    POSTGRES_PASSWORD: sonar_pass
  volumes:
    - sonarqube_db_data:/var/lib/postgresql/data
    - ./docker/sonar-init:/docker-entrypoint-initdb.d
  ports:
    - "5433:5432"      # Puerto diferente a Healenium (5432)
  healthcheck:
    test: ["CMD-SHELL", "pg_isready -U sonar -d sonarqube"]
    interval: 10s
    timeout: 5s
    retries: 10
  networks:
    - sonar-net

sonarqube:
  image: sonarqube:10.4-community
  container_name: sonarqube
  depends_on:
    sonarqube-db:
      condition: service_healthy
  environment:
    SONAR_JDBC_URL: jdbc:postgresql://sonarqube-db:5432/sonarqube
    SONAR_JDBC_USERNAME: sonar
    SONAR_JDBC_PASSWORD: sonar_pass
    SONAR_CE_JAVAOPTS: "-Xmx1g -Xms256m"
    SONAR_WEB_JAVAOPTS: "-Xmx512m -Xms128m"
    SONAR_SEARCH_JAVAOPTS: "-Xmx512m -Xms512m"
  volumes:
    - sonarqube_data:/opt/sonarqube/data
    - sonarqube_extensions:/opt/sonarqube/extensions
    - sonarqube_logs:/opt/sonarqube/logs
  ports:
    - "9000:9000"
  ulimits:
    nofile:
      soft: 65536
      hard: 65536
  networks:
    - sonar-net

🗄️ 5.3 Inicialización de Base de Datos

El archivo docker/sonar-init/01-init-sonarqube.sql se ejecuta automáticamente al crear el contenedor sonarqube-db:

-- Crear base de datos sonarqube
CREATE DATABASE sonarqube
    WITH OWNER = sonar
    ENCODING = 'UTF8';

GRANT ALL PRIVILEGES ON DATABASE sonarqube TO sonar;
ALTER USER sonar WITH SUPERUSER;

🚀 5.4 Levantamiento del Servicio SonarQube

  1. Configurar vm.max_map_count (solo primera vez en Windows) 
    # PowerShell como Administrador
wsl -d docker-desktop sysctl -w vm.max_map_count=262144
  2. Iniciar solo los servicios SonarQube 
    # Solo SonarQube (sin Healenium)
docker compose up sonarqube-db sonarqube -d

# O todos los servicios
docker compose up -d
  3. Esperar a que SonarQube esté listo (~2-3 minutos) 
    # Ver logs en tiempo real
docker compose logs -f sonarqube

# O verificar el healthcheck
docker compose ps sonarqube
# Esperar status: healthy
  4. Acceder al portal web

    Abrir en el navegador: http://localhost:9000

    Credenciales por defecto:

    • Usuario: admin
    • Contraseña: admin
    🔐

    SonarQube pedirá cambiar la contraseña en el primer login. Cambiarla por una segura.

  5. Generar token de análisis

    En el portal: My Account → Security → Generate Tokens

    Tipo: Global Analysis Token o Project Analysis Token

    Guardar el token generado — solo se muestra una vez.

Reportes y Quality Gates

📈 7.1 Reportes HTML Generados

Cada herramienta genera un reporte HTML interactivo en build/reports/. Los reportes incluyen:

  • Checkstyle: Lista de violaciones de estilo con enlace al archivo y línea
  • PMD: Violaciones por categoría con descripción y prioridad
  • SpotBugs: Bugs por categoría (CORRECTNESS, PERFORMANCE, SECURITY) con stack trace

🚦 7.2 Configuración del Quality Gate en SonarQube

El Quality Gate es el semáforo que decide si el análisis "pasa" o "falla" según umbrales definidos. Para configurarlo en SonarQube:

  1. Ir a Quality Gates en el portal

    Menú: Quality Gates → Create → Nombre: "Telefónica Gate"

  2. Configurar condiciones recomendadas para nuevo código:
    Métrica Operador Umbral
    New Bugs is greater than 0
    New Vulnerabilities is greater than 0
    New Security Hotspots Reviewed is less than 100%
    New Code Smells is greater than 10
    New Duplicated Lines (%) is greater than 3%
  3. Asignar el Quality Gate al proyecto

    En el proyecto: Project Settings → Quality Gate → Select "Telefónica Gate"

Integración CI/CD

Para integrar el análisis estático en un pipeline CI/CD (Jenkins, GitLab CI, GitHub Actions):

🔄 GitHub Actions

# .github/workflows/static-analysis.yml
name: Static Analysis

on:
  push:
    branches: [ master ]
  pull_request:
    branches: [ master ]

jobs:
  analyze:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Set up JDK 21
        uses: actions/setup-java@v4
        with:
          java-version: '21'
          distribution: 'temurin'

      - name: Run Static Analysis
        run: ./gradlew staticAnalysis

      - name: SonarQube Analysis
        run: ./gradlew sonarqube
        env:
          SONAR_TOKEN: ${{ secrets.SONAR_TOKEN }}
          SONAR_HOST_URL: ${{ secrets.SONAR_HOST_URL }}

      - name: Upload Reports
        uses: actions/upload-artifact@v4
        if: always()
        with:
          name: static-analysis-reports
          path: build/reports/

🏗️ Jenkins Pipeline

// Jenkinsfile
pipeline {
    agent any
    tools { jdk 'JDK-21' }

    stages {
        stage('Checkout')  { steps { checkout scm } }

        stage('Static Analysis') {
            steps {
                sh './gradlew staticAnalysis'
            }
            post {
                always {
                    recordIssues tools: [checkStyle(pattern: '**/checkstyle/*.xml'),
                                         pmdParser(pattern: '**/pmd/*.xml'),
                                         spotBugs(pattern: '**/spotbugs/*.xml')]
                }
            }
        }

        stage('SonarQube') {
            steps {
                withSonarQubeEnv('SonarQube') {
                    sh './gradlew sonarqube'
                }
            }
        }

        stage('Quality Gate') {
            steps {
                timeout(time: 5, unit: 'MINUTES') {
                    waitForQualityGate abortPipeline: true
                }
            }
        }
    }
}

Resolución de Problemas

Problema Causa Solución
vm.max_map_count too low SonarQube/ES necesita valor alto wsl -d docker-desktop sysctl -w vm.max_map_count=262144
SonarQube no arranca (OOM) Poca memoria asignada a Docker Docker Desktop → Settings → Resources → Memory: mínimo 6 GB
Checkstyle: maxWarnings exceeded Hay violations de estilo Revisar build/reports/checkstyle/main.html y corregir
PMD build failed Violaciones de reglas PMD Revisar build/reports/pmd/main.html; ajustar código o ruleset
SpotBugs falla con StackOverflow Análisis recursivo profundo Añadir -Xss2m a spotbugs.jvmArgs
401 Unauthorized en Sonar Token inválido o expirado Regenerar token en My Account → Security → Tokens
Puerto 9000 ocupado Otro servicio usa el puerto Cambiar en docker-compose: "9001:9000" y actualizar sonar-project.properties
Checkstyle falla en clases Lombok Código generado detectado Añadir entrada en config/checkstyle/suppressions.xml

# ── ANÁLISIS LOCAL (sin Docker) ───────────────────────────
.\gradlew.bat staticAnalysis              # Todo el análisis estático
.\gradlew.bat checkstyleMain              # Solo estilo
.\gradlew.bat pmdMain                     # Solo PMD
.\gradlew.bat spotbugsMain                # Solo SpotBugs

# ── DOCKER ───────────────────────────────────────────────
docker compose up sonarqube-db sonarqube -d   # Levantar SonarQube
docker compose down                           # Parar todo
docker compose down -v                        # Parar + borrar volúmenes

# ── SONARQUBE ─────────────────────────────────────────────
.\gradlew.bat sonarqube -Dsonar.token=TOKEN   # Enviar análisis
# URL: http://localhost:9000
# Credenciales iniciales: admin / admin
